Introduction

OpenSourceMark is a powerful, far reaching tool useful for obtaining valuable computer performance insight. Contrasting with commercial benchmarks like SysMark or PCMark, OpenSourceMark is completely Open Source leading to unmatched verifiability.

There are many computer benchmarks out there, but very few match the scope, power and polish of OpenSourceMark.  Those benchmarks that do approach OpenSourceMark are most often commercial, closed source efforts.

 

Unfortunately, it has been demonstrated time and time again that commercial, closed source benchmarks are untrustworthy.  The financial motivations for microprocessor vendors to “influence” commercial benchmarks are great.  When a new x86 CPU debuts, billions of dollars can be at stake.  A poor showing on a critical benchmark could expose a design weakness that might cost a processor company a vast amount of momentum, money and time.

 

It has been shown that large benchmark “consortiums” like BAPCo are often little more than marketing devices for patron companies like Intel.  Inside the computer industry, it is also known that seemingly independent, high profile, commercial benchmark corporations are allowing certain CPU firms to secretly write their tests.

 

There are infinite ways to bias a benchmark to make one processor look better than all the others.  Ultimately, it doesn’t matter who produces the benchmark, but what matters is how verifiable that benchmark is.  If a benchmark cannot be verified as trustworthy, it is worse than useless to the objective user, it is a waste of time. 

 

An unverifiable benchmark is like a loaded and cocked pistol sewn up inside a pillow: you never know what you’re going to shoot or when you’re accidentally going to shoot yourself.  The only sensible way for a benchmark to be truly verifiable is to take the pistol out of the pillow and make the benchmark Open Source.

 

OpenSourceMark, the flagship project for the Comprehensive Open Source Benchmarking Initiative, or “COSBI” for short, is intended to be open ended, flexible and, most importantly, verifiable.

 

The OpenSourceMark project is currently focusing on Windows XP.  There is a much more primitive version of OpenSourceMark available for Linux and we intend to eventually bring Linux support on par with the Windows version.

 

Borland Delphi 2005 Professional is the programming language for the test harness.  Most of the tests are also written in Delphi 2005.  However, many OpenSourceMark tests are written in C and C++ and compiled in Microsoft Visual C++ or g++ (the excellent Dev-C++ mingw environment).

Objectives

The key objectives for the COSBI OpenSourceMark project are:

o       Verifiable: All of the source code for OpenSourceMark is freely available in order that every aspect of the program can be exhaustively scrutinized by anyone. The entire development project is conducted on a public message board, so that anybody can participate.

Tip: The 800-pound gorilla in the commercial benchmarking jungle is BAPCo, a company that has been shown in the past to bias its tests in favor of Intel. This shouldn't be much of a surprise since Intel has quietly written much of BAPCo's PC centric benchmarking code. In fact, for many years BAPCo's mailing address was the same as Intel's.

Although we exposed BAPCo several years ago for many of its deplorable practices, the already shady organization now encrypts its intermediate results files in some of its newest benchmarks like SysMark2004SE and MobileMark. On older BAPCo benchmarks like SysMark2002, it was possible to “hack” the benchmark script to expose the text file that the benchmark used to track the intermediate task timings that BAPCo used to generate overall scores. This process allowed us to gain insight into what BAPCo was actually testing. For instance, 90% of the Microsoft Excel component of the SysMark2001 score was composed of the time it took to load the Excel application from disk. The next year, sorting, the Pentium 4's forte and a rare task in Excel, became responsible for over 90% of Excel's contribution to the SysMark2002 score! But instead of BAPCo learning its lessons from its public humiliation and coming clean with the computing industry by opening its benchmark source code and providing direct access to result calculations to the public, BAPCo has merely retreated into even more secrecy and obfuscation. Given BAPCo's past behavior, we can only assume the worst.

o       Meaningful: Commercial benchmarks are notorious for spitting out “dumb numbers.”  A lengthy test suite will run for hours before burping out a single number with no detailed explanation for what exactly is being measured. OpenSourceMark provides dozens of targeted tests with each test providing its own score.  Each Official Run test is described in this document and the source code for all tests can easily be examined. All composite scores are simply the arithmetic average of the individual test results

o       Precise: The tests included with OpenSourceMark are designed to target very specific computational aspects. Consequently, individual test results are easy to interpret and understand.

o       Scalable: Even before the explosion of multicore processors began changing the computing landscape, OpenSourceMark was developed with multicore technologies in mind. A large number of OpenSourceMark tests are threaded and readily show the benefits of the newest multicore CPUs. OpenSourceMark's sophisticated, object-oriented design with fully encapsulated data even takes advantage of high-powered NUMA-based multiprocessing systems.

o       Customizable: OpenSourceMark deploys several different suites of tests that are intended to reflect broad computing characteristics. However, OpenSourceMark is easily customizable so that users can chose which tests are most important to them.

o       Comprehensive: OpenSourceMark hosts a wide variety of tests covering diverse performance aspects. The number of tests will grow rapidly. The COSBI development plan is to be all-inclusive: all submitted tests will be added to OpenSourceMark as long as the tests are not redundant and are demonstrably useful.

o       Useful: In addition to a large and growing number of useful performance tests, OpenSourceMark also includes tools to investigate memory and disk performance, detect CPU performance throttling, chart CPU clock speed, CPU utilization and more.

o       Modern: OpenSourceMark is optimized to use the newest CPU features like SSE2, SSE3 and VIA's advanced Padlock security support.

o       Inclusive: A cornerstone of the OpenSourceMark project is inclusiveness.  All meaningful tests submitted to the OpenSourceMark project that do not duplicate existing tests will be incorporated into OpenSourceMark.

We think you'll find OpenSourceMark to be a powerful and trustworthy program for measuring computer performance, battery life and test system stability. Even at this early development stage, OpenSourceMark is demonstrably one of the most comprehensive and useful benchmarks ever developed.

OpenSourceMark, an open act of rebellion against the powers inside the computing industry that work to control perception, is released under the GNU General Public License as described at the following URL:  http://www.gnu.org/licenses/gpl.html

Understanding Computing Performance

Personal computers are used for a wide variety of tasks. Since no one uses his computer exactly like anyone else, producing a single number to represent overall computing performance is not easy.

A quantity that can be represented by a single number is referred to as “scalar.” Overall computing performance is not scalar, but can be represented by a six-dimensional vector. A “vector” is a quantity that has magnitude and direction, or requires more than one number to describe.

Here is a breakdown across the six major “performance vectors.”

o        3D Gaming: Modern 3d rendered games create highly interactive, visually rich worlds and impose the most computationally intensive burdens commonly carried out on personal computers. Of course, graphics card choice, screen resolution, color depth, driver implementation and enabled rendering features have the biggest impacts on gaming performance. However, floating point throughput (FPU, SSE, SSE2, SSE3, etc,), memory amount, memory latency and memory bandwidth are also very important. For games that are not 3D intensive, responsiveness qualities (see below) are typically most vital. OpenSourceMark Official Run tests stressing this vector are: TalphaBlendTest, TalphaDotsTest, TBandwidthBP64ThreadTest, TfernTest, TlorenzTest, TmandelbrotThreadTest, TmemLatencyThreadTest, TNBodyOpenGL, TnBodyTest, TSse3ThreadTest, TtrigTest, Ttrig2Test.

Tip: OpenSourceMark currently provides a 3d Gaming Performance Vector that primarily shows the CPU's suitability for gaming.  In future versions of OpenSourceMark we will incorporate many more OpenGL and DirectX tests so that the graphics card's contribution to 3d gaming can be better represented.

o        Artificial Intelligence: Gaming logic, financial modeling, expert systems, robotics path finding and some types of image recognition are all examples of artificial intelligence (AI), a unique performance category. Certain spam filters and anti-virus packages also use AI. OpenSourceMark Official Run tests stressing this vector are: TmazeTest, TmemLatencyThreadTest.

Tip: AI is not strongly represented in the current version of OpenSourceMark. We are soliciting for additional tests that fall within this performance vector.

o        Data Set Transforms: Media encoding and decoding, file compression and decompression, image processing and audio mixing are examples of data set transformations. Many people now watch movies on their computers, and growing numbers of people use their PCs to produce motion pictures. With the digital photography revolution, the personal computer has become an invaluable tool for viewing and manipulating family photographs. File compression/decompression is a common task that often closely tracks encoding performance. Memory bandwidth is very important in this domain, but the CPU also needs to be fast enough to handle the incoming data. Superscalar and/or multicore microprocessors are strong in this domain. OpenSourceMark Official Run tests stressing this vector are: T7zipThreadTest, TBandwidthBP64ThreadTest, TidenticalThreadTest, TimageResizeTest, TimageRotateTest, TjpgDecodeTest, TlameEncoderThreadTest, TpngOutThreadTest, TSse3ThreadTest, TupxThreadTest, TzipCompressTest.

o        Math/Science/Engineering: For a small subset of computer users, math, scientific and engineering applications constitute an important slice of the computing pie. While some applications in this vector overlap data set transformations, these programs tend to greatly emphasize floating point performance (including transcendentals). In floating point intensive streaming cases, the memory subsystem needs to be fast enough to keep the FPU/SSE2 units fed. Numerical integration carried out in new physics acceleration add-in cards falls within this vector. OpenSourceMark Official Run tests stressing this vector are: TbandwidthBP64ThreadTest, TdhrystoneThreadTest, TfernTest, TfibThreadTest, TgridBlastFPTest, TlorenzTest, TmandelbrotThreadTest, TmontgomeryMultiplierTest, TNBodyOpenGL, TnBodyTest, TorthogonalThreadTest, TpiTest, TpiThreadTest, TSse3ThreadTest, TtrigTest, Ttrig2Test, TWhetstoneThreadTest.

o        Responsiveness: The most overlooked aspect of computing performance is probably the single most important performance area for general computing. Responsiveness is a measurement of a system's ability to keep up with user input and is characterized by extremely branchy code that typically requires intensive interaction between the CPU, display adapter, memory, and hard drive. A responsive computer is snappy and quick while a computer with poor responsiveness feels sluggish and lethargic. As user interfaces grow more complicated, responsiveness becomes more important. For example, Windows Vista imposes prodigious demands even to render the simplest application windows; many systems will not be able to provide an enjoyable computing experience unless a number of Vista's new interface features are turned off. A system's responsiveness is largely a function of the CPU's integer unit, the CPU's pipeline depth and cache line size, memory latency, memory amount, hard drive speed and interface, the video core's 2d unit, P-State transition latencies and thresholds, screen resolution, refresh rate and color depth and the CPU's interface to the video core. Vista will also make heavy demands on 3D rendering and cryptographic throughput (see Security below). Surprisingly, small, simple, power efficient CPUs can be more responsive than massive, deeply pipelined microprocessors like the Pentium 4. Multicore chips will help responsiveness a lot, especially when antivirus software is running in the background. Thin-clients fall essentially 100% inside this performance vector. Office application suites and web browsing are strongly driven by this vector. AMD Athlon 64 X2 processors are currently the acme of performance for this vector. OpenSourceMark Official Run tests stressing responsiveness are: TFilledCircleTest, TdrawEllipsesTest, TfernTest, TfileCopyTest, TGridBlastFPTest, TgridBlastTest, TdrawLinesTest, TmemLatencyThreadTest, TplotLinesTest, TrandomDotsTest, TdrawRectanglesTest, TrichEditTest, TwebPageLoad.

o        Security: Rapidly growing in importance, security involves everything from protecting online transactions to encrypting file systems and data networks on the fly. Not only is security necessary to lock away information from prying eyes, but movie and music providers are insisting on a cryptographically secure data streams for next generation media. Facilitating this, Windows Vista steps up cryptography demands by vast leaps and bounds by using the CPU to perform AES encryption on media streams on the fly. While most security functions fall within the data set transform domain, implementation concerns place security within its own unique performance vector. OpenSourceMark Official Run tests stressing this vector are: TAesThreadTest, TmontgomeryMultiplierTest, Tsha1Test, Tsha256Test.

In addition to the tools necessary for characterizing a system by the performance vectors listed above, OpenSourceMark also provides benchmarks for measuring notebook performance and battery life.

Using OpenSourceMark

Unzip the OpenSourceMark archive preserving the directory structure. Inside the OSMark directory, double-click on the CosbiOpenSourceMark.exe executable. Below is an image of the OpenSourceMark main window.


Click the Official Run button to kick off a suite of tests that generates an OpenSourceMark official performance score

Tip: You can abort OpenSourceMark tests by repeatedly press the <Esc> key.

Official Run is the benchmark suite that people will use most often and is the only run currently tracked by the Result Viewer. When you click the Official Run button, the Options dialog, discussed below, is first opened before the tests begin.  At the end of the run, which take 30 minutes to an hour to complete, OpenSourceMark's powerful Result Viewer is shown.

Note: All reference scores were obtained on systems running with video resolutions of 1024x768x32 @ 75Hz (60Hz for notebooks).  All Official Run scores are normalized to 1,000 against a 3.4GHz Intel Pentium 4 ™ with 512MB of RAM and using a GeForce FX 5600 ™.  The two exceptions are the Encrypt/Decrypt test, which is normalized against a 1.333GHz VIA C3 ™, and the SSE3 test, which is normalized against a 1.8GHz AMD Turion 64 ™.  Active antivirus software can severely undermine performance on several tests, so all reference scores were obtained without active antivirus software.



As with Official Run, a click on the Custom Run... button first opens the Options dialog which is explained below. After the Options dialog settings are approved, you'll see a new window where you can select any combination of individual tests to run.



You can also specify the number of iterations to loop over your collection of tests. In a multitasking environment like Windows XP, it is best to choose three or more iterations just in case background tasks occur that perturb test runs. After you click “OK,” your selections will execute.

When the tests complete, OpenSourceMark will produce a list of the best run for each test along with an average over the best run list.



OpenSourceMark now features four different benchmarks specifically designed for notebook computers. It is very important to prepare your notebook properly for these tests. Fortunately, preparation is easy to do: just follow the brief, simple instructions shown in the Mobile Test dialog before starting a run.


As the tests run, OpenSourceMark will monitor battery levels and will write out a result file into the OSMark subdirectory (or wherever you specify) and shut down the notebook when the battery charge falls to 3%.

Commercial mobile benchmarks are notoriously meaningless. We've tried to create realistic, easy to understand conditions that have obvious merit. For instance, you can be sure that your notebook's battery life will fall between the Maximum Power Draw and Minimum Power Draw corner cases. If you have time for only one mobile benchmark, then Normal is a good compromise between the two.

If OpenSourceMark detects a battery in the system, OSMark will display a battery meter that is handy when testing notebook battery life:


Here is a list of the four different mobile benchmarks:

o        Normal: This mobile benchmark will first execute an Official Run in order to generate a performance score. This will then be followed by a battery depletion period where individual tests are run with one minute idle times in between each test. The battery depletion period will continue until battery levels reach 3% at which point OpenSourceMark will write out a results file and shut down the notebook. The Normal test produces three scores: Performance, Battery Life, and Overall. The Overall score is the arithmetic average of the Performance and Battery Life scores. Note: Older notebooks or notebooks poorly optimized for battery life might not be able to complete the performance portion of the benchmark before the battery is depleted.

o        Maximum Power Draw: The only differences between the Maximum Power Draw and Normal tests is that all of the idle periods have been removed for the former. Additionally, “Spin-up” is enforced between each performance test, forcing the CPU to run at its highest P-State frequency before each test is launched. Note: It is even harder for older, battery sucking notebooks to complete the performance portion of the Maximum Power Draw test than for the Normal test. Because so many older notebooks will struggle on the Max Power test, we strongly recommend that all notebooks be run with the screen set to turn off after one minute as described on the Mobile Test dialog.

o        Minimum Power Draw: Under this condition, the system sits idle until the battery is depleted. Only battery life is tracked in the Minimum Power Draw test. As with the other Mobile tests, a result file is created before OpenSourceMark shuts down the system.

o        PCWorldBench5: One of the few commercial benchmarks that does not stink to high heaven with corruption is PCWorldBench5. If you have this benchmark installed on your system, OpenSourceMark can use the Microsoft Office portion of PCWorldBench5 as load in its mobile test. However, setting this up is a little tricky. In PCWorldBench5, you'll need to launch the MS Office test, but kill PCWorldBench5 through Task Manager as soon as the Office test starts. This should leave the system with all of the necessary applications and scripts that are needed by OpenSourceMark. The results file will contain both battery life and number of iterations completed. The iteration count indicates how much work the system was able to complete before the battery was drained.

A thin client is a PC or other computing device that behaves as a terminal connected over a network or phone line to a server. Thin clients have become extremely popular especially in businesses because thin clients are typically cheaper than full-blown PCs, and, more importantly, the costs to administer thin clients is much lower than the costs to maintain a fleet of PCs. Administration costs for thin clients are so low because only a few servers need to be tended instead of hundreds or thousands of PCs.

The modern thin client user experience is almost exactly the same as working on a PC. You'll see the same familiar Windows or Linux or even Macintosh desktop and applications all behave the same as if they are running locally.

However, there are good and bad thin clients. Although much of the computational work is done on the server, the thin client still must be fast enough to render complicated, graphically rich user interfaces. Unfortunately, some low end thin clients are insanely, maddeningly slow. In fact many thin clients are so slow that data is easily lost because interface rendering can grossly lag user input.

The OpenSourceMark thin client benchmark contains a suite of tests that are specifically made for evaluating thin client performance.  The Thin Client Run is the first suite of tests anywhere that takes into account the effects of the client side buffering that can invalidate using PC centric 2d rendering tests.

Take, for instance, RDP (Microsoft’s Remote Desktop Protocol) based devices where the thin client serves, from the server’s point of view, as essentially a graphics adapter.  The thin client absorbs a stream of GDI commands from the server and renders them on the client.  To expedite processing, the thin client has a buffer that can absorb a large, but finite number of rendering commands.  If the GDI operations are complex but few, the thin client’s buffer can absorb them all in a fraction of the time that it takes the thin client to render them.  However, from the server’s vantage point it thinks the rendering process is finished as soon as it sends out its last operation even though the thin client might take minutes more to draw the screen.  A benchmark under these conditions would register an inordinately high and invalid score.

To overcome this problem, OpenSourceMark uses a new TThinClientTest base class that follows up every test with a buffer flooding stream of very simple plot pixel commands that are rendered almost instantaneously.  Because the buffer is flooded, this last, quick rendering stream forces the server to wait until the test is truly completed.

To run the thin client tests, connect to the Windows server from the thin client to be tested.  Through the thin client, launch OpenSourceMark on the server and then simply click Thin Client Run.

Executing the thin client test natively on Windows XP or XP Embedded systems is also useful for evaluating thin client performance.



The Options button opens the run options dialog. “Run Options” and “Drive target” settings are persistent across OpenSourceMark sessions.


In order to drive the microprocessor to its highest P-State frequency, “Spin up” deploys a load immediately before each benchmark.

We recommend that you check “Defragment drive” and “Process Idle Tasks” whenever you are serious about getting publishable results.

Changing Program Priority is rarely necessary. If you check “Real-Time”, you will find that OpenSourceMark will hog the CPU, making other programs running on your system extremely unresponsive.

The default test timer uses the Windows performance counter. Occasionally, this timer is not reliable. You can then select either “Low,” which is based on GetTickCount, or “CPU” which is a timer class wrapper around TSC (the CPU time stamp counter). “CPU” can be problematic on systems that use P-States and/or C-States, although this situation will change with Microsoft's recent direction to CPU vendors requiring that the TSC update frequency remain constant across P-State and C-State transitions.

You can zoom in on specific tests, filter out systems that you don't care to look at, import and export data, change the chart background image, print the chart, save the chart to a file, edit the data and much more.




The Help button opens the file you are currently reading, “OSMarkDocumentation,html,” in the default web browser.

The Quit buttons exits OpenSourceMark.

If you'd like to see the credits for OpenSourceMark, click the About button.

Stability Test will kick off 500 iterations of the Official Run. On slow systems, this could take a month or more to complete. However, it is not necessary to let the program run for this long. If your computer can churn through Stability Test for a day or so, then you can rest assured that your system is stable.

Remember, you can always abort OpenSourceMark test by repeatedly pressing the <Esc> key.

ThreadWrecker spawns eight extremely demanding threads and will time how long it takes your system to complete them all. If your system is unstable, ThreadWrecker may cause it to crash.

WhetBurn is a utility for detecting thermally induced CPU performance throttling. You can select three different algorithms to apply as loads as well as tweak “n,” the number of Whetstone loops, the Fibonacci sequence number or the number of pi digits to calculate. Once you click “Burn!” to begin the test, WhetBurn will iterate over your load and chart the amount of time that each iteration takes to complete. If the CPU begins to overheat and apply thermal throttling to avoid burnout, the time to complete an iteration will increase which will be easily seen on the chart.

Be sure to set “Iterations” high enough to allow the CPU time to heat up. An overall run time of 30 minutes to an hour is usually sufficient.



You can save the chart or the raw data to a file. You can also print the chart.



BandwidthBurn was the first PC benchmark to graph bandwidth over increasingly large data set sizes. This technique will graphically expose the presence of L1d, L2 and L3 caches. The chart below shows the P4-Northwood's tiny 8kB L1-d and its 512kB inclusive L2. Main memory bandwidth is strong at over 2GB/s. The P4 chart below is noisy because there were lots of background activity running on that machine.



With the “Test” option, you can select which assignment method to use in the bandwidth tests. “Simple” and “Block Prefetch64” are two good conditions to examine.

You can zoom in to take a closer look at data by click and dragging a bounding rectangle from upper-left to lower-right. Reversing the direction will unzoom the chart.

Make sure that “Max Dataset (kB)” is large enough to encompass enough main memory to ensure that cache effects are eliminated. It might be necessary to set this value as high as twice the size of the cacheable region.

Because of the tiny amounts of time that BandwidthBurn measures, it must use TSC as a timer. On notebooks it will often be necessary to check the “Spin-Up CPU” box to ensure that the system is running at the highest P-State frequency before starting tests.

MemLatencyPlus is a unique benchmarking tool that measures a computer's speed at randomized memory accesses. Although the HPCChallenge RandomAccess tests exercises a condition very similar to MemLatencyPlus, MemLatencyPlus predated the HPC tests was the first benchmark to measure randomized memory access speeds. PCMark also includes random access memory tests, but they were added at our request based upon experience with MemLatencyPlus.

Modern applications running on today's operating systems with complicated GUIs and component based architectures, are often extremely branchy and touch data structures that are essentially in random memory locations. Because of this, MemLatencyPlus is a very good predictor of performance on applications like MS Office, OpenOffice and many common day to day computing tasks.



“Test Memory Latency” is a quick and dirty benchmark of random memory access performance on a 16MB data set, while “Graph Memory Latency” charts performance out to 4MB.

OpenSourceMark devotes large portion of it main window to output results details.  All of this editable data can be dumped to a file by clicking “Save.” The data can also be quickly copied to the clipboard by clicking “Copy to Clipboard.”



If OpenSourceMark detects a battery in the system, it will display a battery meter that is handy when testing notebook battery life.

Command Line Options

OpenSourceMark supports command line arguments.  This facility allows tests to be scripted.  For example, if you wanted to defrag, process idle tasks, execute three Official Runs, show the status window between tests, save the result output to the default filename and exit when the tests complete, the following command line would do the trick.

CosbiOpenSourceMark or i 3 e s pit defrag ss
